# Медиа-инструменты

Инструменты для компиляции разных видов медиа, почти полностью вайбкод и AI-generated.  

Проект делался как эксперимент:  
**может ли AI сделать всё сам?**

## Описание проекта

Это набор медиа-инструментов.

### Состав

**Video** — позволяет добавлять дорожки аудио и субтитров к видео пакетно.

**Track** — позволяет к аудиофайлу mp3 добавлять metadata: автора, название, теги, жанр и т.д.

**Image** — позволяет конвертировать изображения в разные форматы.

**PDF** — позволяет редактировать страницы PDF: добавлять, удалять, менять порядок, разделять на JPEG-изображения, а также сжимать PDF-файлы.

## Используемая модель
- Claude Haiku 4.5 через Copilot, встроенный в VSCode.
- GPT-5.4 через Codex.

## Как пользоваться

Всё максимально просто (веб-приложение на Flask, все модули запускаются локально):

1. Клонируем репозиторий.
2. Создаём виртуальное окружение.
3. Устанавливаем зависимости:

```bash
pip install -r requirements.txt
```

4. (Опционально) Задаем переменную окружения `VIDEO_ROOT` для browser-режима
```bash
export VIDEO_ROOT=/path/to/video/for/compile
```
> В десктопном режиме (`desktop.py`) переменная `VIDEO_ROOT` не требуется, так как пути к файлам выбираются через диалоги.
> В browser-режиме (`main.py`) переменная нужна для поиска аудио дорожек и субтитров, например:
> - `C:\Users\*USERNAME*\Videos`
> - `/Users/*USERNAME*/Movies`

> Для работы модулей **Video** и **Track** требуется установленный `ffmpeg` и запуск на локальной машине с доступом к исходным файлам.

5. Для browser/dev-режима запускаем:

```bash
python3 ./main.py
```

6. Для desktop-режима запускаем:

```bash
python3 ./desktop.py
```

## Режимы запуска

### Desktop

`desktop.py` поднимает Flask как локальный встроенный backend и открывает интерфейс в окне приложения через `pywebview`, без системного браузера.

### Browser

`main.py` остаётся удобной точкой входа для разработки: стартует локальный сервер и открывает интерфейс в браузере.

## Переменные окружения

Поддерживаются дополнительные переменные:

```bash
export APP_MODE=browser   # или desktop
export APP_HOST=127.0.0.1
export APP_PORT=5000      # если порт занят, приложение возьмёт свободный
```

Если указать `APP_PORT=0`, приложение всегда будет выбирать случайный свободный порт.

## Сборка desktop-приложения

Сборка через PyInstaller:

```bash
pyinstaller media_tools.spec
```

Результат по умолчанию:
- Linux/Windows: `dist/media-tools`
- macOS: `dist/Media Tools.app` и `dist/media-tools`

### macOS: `.app` и `.dmg`

На macOS можно собрать сразу и приложение, и установочный образ:

```bash
chmod +x scripts/build_macos.sh
./scripts/build_macos.sh
```

После этого появятся:
- `dist/Media Tools.app`
- `dist/media-tools-macos.dmg`

Для сборки `.dmg` используется системная утилита `hdiutil`, поэтому этот шаг работает только на macOS.

В spec-файле уже включены:
- Flask templates
- Flask static assets
- hidden imports для `pywebview`

## Внешние зависимости

- `ffmpeg` должен быть установлен и доступен в `PATH`
- `VIDEO_ROOT` нужен для модуля `Video` в browser-режиме (в десктопном режиме не требуется)
- на macOS и Windows права доступа к файлам и папкам могут влиять на работу выбора/обработки медиа

## Наблюдения

### 1. Скорость разработки  
Это реально ускоряет разработку.  
Для прототипа — отличный инструмент.  
Поднять рабочую версию за пару вечеров — вполне реально.

### 2. Повторяющийся дизайн  
Если нужен одинаковый UI — лучше заранее сделать основу самому или навайбкодить базовый шаблон.  
Иначе AI начинает изобретать «то же самое, но другое».  
Вроде похоже, но всё отличается.

### 3. Фронтенд  

Я не любитель разрабатывать фронтенд сам, и здесь AI меня разочаровал.

Он не видит интерфейс.  
Там, где человек видит косяк или неудобство, AI видит, что ты «забыл стиль».

Был показательный момент: проблема была в одном месте,  
а AI упорно пытался чинить совсем другое.

То есть вместо того чтобы понять причину, он начинал править симптомы — причём не те.

С PDF пришлось повозиться — если что-то съехало, AI не может это увидеть.  
А по подробному текстовому описанию проблемы он не всегда способен её диагностировать.

При этом, на удивление, сложный интерфейс Video сделал с первого раза, соблюдая запросы.  
Иногда он просто попадает.

Для простых фронтов (Track, Image) — вполне пригоден.

### 4. Бекенд  

Рабочий.  
С прототипированием справляется хорошо.

К чистоте кода и соблюдению архитектурных концепций есть вопросы.  
Но если цель — быстро получить рабочий результат, он это делает.
